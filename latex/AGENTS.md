## Routing-accuracy methodology (critical — do not regress)

The paper evaluates routing accuracy against **two independently-defined literature-cited oracles** (§VIII.B "Routing Accuracy"). This replaced an earlier unsubstantiated "100% accuracy" claim that a reviewer would have rejected immediately.

**Never** reintroduce phrases like:
- "appropriate routing for all 100 workloads"
- "correct routing decisions across all workload categories"
- "100% routing accuracy"

Report the oracle-based numbers instead. Current honest numbers (as of the last run):

| Oracle | Accuracy | F1(Q) | Recall(Q) | Cohen's κ |
|---|---|---|---|---|
| A: Algorithmic Advantage | 0.810 | 0.457 | 0.571 | 0.382 |
| B: Hardware Viability | 0.860 | 0.500 | **1.000** | 0.477 |

### The two oracles

1. **Oracle A (Algorithmic Advantage):** labels Quantum when a published quantum algorithm provides theoretical speedup. Cites Preskill 2018, Shor 1997, Grover 1996, Kandala 2017, Farhi 2014, Arute 2019.
2. **Oracle B (Hardware Viability):** labels Quantum only when current NISQ hardware produces usable signal. Thresholds calibrated to the hardware-validation table (currently Table XI, 27-circuit `ibm_fez` validation — *label* is `tab:hardware-validation`, always cite by label never by Roman numeral): post-transpile depth ≤ 200, sub-20-qubit classical, 20–50-qubit entangled circuits quantum, etc.

Both oracles apply **only to the workload filename and physical circuit features (qubits, depth, entanglement)** — they do not touch Nexar's rule engine, ML model, or cost analyser. This independence is what makes the evaluation defensible.

## Rule-engine invariants referenced by the paper

Two rule-engine behaviours are claimed explicitly in §V, §VIII, and the Case Study, and **must be preserved** in `decision-engine/services/rule_service.py`:

1. **Physical-necessity FORCE_QUANTUM** for `50 ≤ qubits ≤ 127`. Classical state-vector simulation is infeasible above 50 qubits (paper §V.A, §VIII.D), so quantum is the only viable path regardless of NISQ fidelity. This fix lets the 100-qubit random circuit and 120-qubit VQE route correctly.
2. **Physical-necessity REJECT** for `qubits > 127`. Exceeds IBM Eagle capacity; classical simulation also infeasible. Enables the 150-qubit oversize test case referenced in the Routing Distribution paragraph.

Both overrides live at the top of `RuleBasedSystem.evaluate()` and precede all other rule paths.

## Style and length conventions

- **Avoid** itemized lists where a compact prose sentence would work. Tight prose saves ½–1 column per list.
- **Avoid** detailed ML hyperparameter enumerations in the main text (dims, heads, vocab size). Put one-line summaries with inline values instead.
- **Avoid** decorative `tcolorbox` output-examples (they consume disproportionate space). Use a short prose case study instead.
- **Cross-references:** use `\ref{sec:routing-accuracy}`, `\ref{sec:cost-break-even}`, `\ref{sec:future-work}`, `\ref{sec:dual-oracle}` — these labels already exist. Keep them stable.
- **Tables**: use `\footnotesize` or `\scriptsize`. Numbers are reported to 3 decimals. Quantum-class metrics use `(Q)` notation.
- **Citations**: IEEEtran bibliography entries are ordered by first citation. When adding a new reference, append at the end — do not renumber.

## Common pitfalls

1. **Stale PDF cache in tooling:** After editing the `.tex`, always force a clean compile (`Remove-Item *.aux, *.log, *.out, *.pdf; pdflatex nexar-paper.tex; pdflatex nexar-paper.tex`) before trusting any extracted text.
2. **Table labels collide with `\ref` targets** — our labels use `tab:` prefix, figures use `fig:`, sections use `sec:`. Do not cross-prefix.
3. **`routing_accuracy_table.tex` is wider than a column** and uses `table*` (2-column span). Do not wrap it inside `table` or the cross-reference will break. The full-width layout is deliberate: the table has 11 columns (1 router + 5 metrics × 2 oracles) and would require illegible font or dropped columns to fit a single column. `table*` is standard IEEE practice for wide result tables and does not violate any conference formatting rule. Do not "fix" it by splitting into two single-column tables — that would separate the Oracle-A/Oracle-B side-by-side comparison which is the table's point.
4. **The abstract is one long sentence.** A reviewer noted this; leave a TODO for a future revision if asked to tighten the abstract.
5. **Do not edit `oracle_ground_truth.csv` by hand.** Every label is justified by the rule cited in the adjacent column; hand-edits break auditability. Regenerate via the script instead.
