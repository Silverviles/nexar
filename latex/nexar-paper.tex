\begin{abstract}
Determining whether a computational workload benefits from quantum or classical execution remains a key challenge in the current era of noisy intermediate-scale quantum computing. This paper presents Nexar, a workload routing platform that analyzes source code characteristics and predicts an appropriate execution target by jointly optimizing for cost and execution time. The system employs a hybrid decision framework that combines a machine learning ensemble trained on a sixteen-dimensional feature vector capturing circuit complexity, entanglement properties, and hardware viability metrics, a physics-aware rule-based engine that enforces hardware constraints and applies algorithm-specific routing logic, and a real-time cost analyzer that estimates execution time and monetary cost across multiple quantum providers and classical computing resources. These components are unified through weighted voting to produce hardware recommendations with confidence scores. The platform incorporates automated code analysis, including algorithm classification with the CodeBERT model and quantum circuit characterization to extract workload features without manual annotation. We evaluate the system across one hundred diverse workload types drawn from search, factorization, optimization, and simulation problem families, and we demonstrate that the framework consistently identifies cost-effective and time-efficient execution targets while serving as a practical decision-support tool for hybrid quantum-classical computing environments.
\end{abstract}

\section{Introduction}

The advent of Noisy Intermediate-Scale Quantum (NISQ) devices \cite{preskill2018} has created a significant challenge in computational resource allocation: given a workload, should it execute on classical hardware, a quantum simulator, or a real quantum device? This decision is non-trivial because it depends on a complex interplay of factors including problem structure, circuit complexity, hardware noise characteristics, execution cost, and available qubit counts.

Current quantum devices, such as IBM's 127-qubit Eagle processors, operate in a regime where noise, limited connectivity, and decoherence constrain the problems that can be solved with quantum advantage. Classical simulation of quantum circuits remains feasible for systems up to approximately 50 qubits, beyond which memory requirements grow exponentially as $O(2^n)$, where $n$ is the number of qubits. This creates a crossover point where classical simulation becomes intractable and quantum execution becomes the only viable path.

However, simply having a quantum device does not guarantee advantage. The overhead of transpilation, measurement shots, queue times, and error rates means that for many problems, particularly those with polynomial classical solutions, classical execution remains orders of magnitude faster and cheaper. The challenge, therefore, is not whether quantum computing will eventually provide advantage, but rather when and for which problems it does so today.

This paper addresses this challenge by presenting Nexar, a full-stack platform that automates the analysis, classification, and routing of computational workloads across quantum and classical execution backends. Nexar's contributions are:

\begin{enumerate}
    \item \textbf{Automated Code Analysis Pipeline:} A multi-dimensional analysis engine that detects programming languages across five quantum and classical frameworks, computes complexity metrics, and classifies algorithms using a three-tier approach combining CodeBERT transformers, ensemble machine learning, and rule-based pattern matching.
    \item \textbf{Hybrid Decision Engine:} A three-component decision framework that merges machine learning (ML) predictions (50\% weight), physics-aware rules (35\% weight), and real-time cost analysis (15\% weight) through weighted voting to produce hardware recommendations with calibrated confidence scores.
    \item \textbf{Provider-Agnostic Execution Layer:} A hardware abstraction layer supporting IBM Qiskit, AWS Braket, and Azure Quantum through a unified Application Programming Interface (API), enabling transparent workload routing across heterogeneous quantum backends.
    \item \textbf{Empirical Evaluation:} A comprehensive benchmark across 100 workloads spanning classical computation, hybrid quantum-classical algorithms, and pure quantum circuits, with cost and performance analysis demonstrating the system's routing accuracy and the current state of quantum-classical cost tradeoffs.
\end{enumerate}

\section{Related Work}

\subsection{Quantum-Classical Hybrid Frameworks}

Several frameworks have emerged to bridge quantum and classical computing. Weder et al.\ \cite{weder2021} proposed automated quantum hardware selection for quantum workflows. Qiskit Runtime \cite{ibm2023} provides session-based quantum execution with classical co-processing. Amazon Braket \cite{aws2023} offers a multi-provider interface but lacks automated workload analysis. Azure Quantum \cite{microsoft2023} provides resource estimation but does not perform code-level analysis for routing decisions. Tang et al.\ \cite{tang2024} proposed AlphaRouter, applying reinforcement learning and tree search to quantum circuit routing. Zhan et al.\ \cite{zhan2025} presented a full-stack framework for high-performance quantum-classical computing. Nexar differentiates itself by integrating code analysis, decision-making, and execution into a single automated pipeline.

\emph{Positioning for empirical comparison.} These systems target different layers of the quantum-classical stack and do not admit a head-to-head comparison with Nexar. AlphaRouter \cite{tang2024} operates at the physical-qubit mapping layer, selecting a coupling-aware placement for a circuit whose execution target is \emph{already} assumed to be quantum. Qiskit Runtime \cite{ibm2023}, Amazon Braket \cite{aws2023}, and Azure Quantum \cite{microsoft2023} are execution platforms that accept whatever the user submits without recommending a quantum-versus-classical split, so their effective ``routing'' is simply the user's submission choice. Weder et al.\ \cite{weder2021} select \emph{among} quantum backends for workflows that have already been classified as quantum. None of these systems answer Nexar's core question---``given arbitrary source code, should this run quantum or classical?''---so our empirical evaluation (Section~\ref{sec:routing-accuracy}) compares against the closest available reference points: a qubit-threshold heuristic, a circuit-volume heuristic, a 1{,}000-seed random router, and the three Nexar components in isolation. The rules-only ablation subsumes the Weder-style rule-based selection logic applied to our 16-feature vector.

\subsection{Quantum Resource Estimation}

Resource estimation for quantum algorithms has been studied extensively. Lammers et al.\ \cite{lammers2025} addressed quantum resource management challenges in the NISQ era \cite{hai2025}. Beverland et al.\ \cite{beverland2022} developed methods for estimating logical qubit requirements for fault-tolerant algorithms. Quetschlich et al.\ \cite{quetschlich2024} demonstrated practical resource estimation for quantum application development. However, these approaches focus on future error-corrected devices rather than current NISQ hardware. Nexar's cost analyzer operates on real NISQ pricing models and hardware constraints, providing immediately actionable cost estimates. Recent work by Kashif et al.\ \cite{kashif2026} has explored hardware-calibrated quantum cost modeling for resource-aware optimization.

\subsection{Algorithm Classification and Code Analysis}

Static code analysis for quantum programs has been explored in works such as ScaffCC \cite{javadiabhari2015} and Q\# compiler toolchains. CodeBERT \cite{feng2020}, a pre-trained model for programming language understanding, has shown strong performance on code classification tasks. Afrose et al.\ \cite{afrose2025} further applied CodeBERT for translating QASM to QIR. Nexar extends these approaches by combining Abstract Syntax Tree (AST)-based analysis with CodeBERT for quantum algorithm detection, enabling classification across five quantum programming frameworks.

\subsection{Quantum Advantage Thresholds}

The question of when quantum advantage emerges has been studied through computational complexity theory \cite{aaronson2011} and empirical benchmarks \cite{arute2019}. Our work contributes empirical data on the physical-feasibility crossover: 50--60 qubits is where classical state-vector memory (16 PB to 16 EB) exceeds any deployable system, making quantum the only viable path for exact simulation beyond this scale.

\subsection{Architecture Overview}

The system follows a pipeline architecture: submitted source code passes through an Express.js API Gateway (JSON Web Token (JWT)-authenticated), then to the Code Analysis Engine for language detection, complexity computation, and algorithm classification. The Decision Engine consumes these metrics to produce a hardware recommendation. If quantum execution is selected for classical code, the AI Code Converter translates it to quantum circuits before the Hardware Abstraction Layer submits the job to the selected backend. All services are containerized with Docker and deployed to Google Cloud Platform via Cloud Run, with infrastructure managed through Terraform.

\subsection{Language Detection}

The engine supports five programming languages and frameworks: Python, Qiskit, Cirq, Q\#, and OpenQASM. \subsection{Algorithm Classification}

Algorithm detection employs a three-tier approach:

\begin{enumerate}
    \item \textbf{CodeBERT Transformer:} A fine-tuned CodeBERT model classifies the algorithm family (e.g., search, optimization, factorization, simulation) based on code semantics \cite{feng2020}.
    \item \textbf{ML Ensemble Classifier:} A scikit-learn ensemble (Random Forest, XGBoost, Gradient Boosting) provides a secondary classification using 35 hand-engineered circuit features (gate counts per type, CX-ratio, depth, rotation-gate distribution, etc.).
    \item \textbf{Rule-Based Pattern Matching:} Deterministic rules identify known quantum patterns (e.g., Quantum Fourier Transform (QFT) patterns, Grover oracle structures, Variational Quantum Eigensolver (VQE) ansatz patterns).
\end{enumerate}

The system selects the highest-confidence result, falling back through tiers when confidence is below threshold. Empirical performance justifies this layering. The tier-1 CodeBERT classifier attains macro F1 of 0.66 across eight algorithm families on a 750-sample held-out set, achieving perfect recognition (F1\,=\,1.00) of Grover, QAOA, QPE, and Shor while missing Bernstein--Vazirani and Deutsch--Jozsa entirely (F1\,=\,0)---oracle-family circuits whose Python surface syntax carries insufficient semantic signal for a transformer pre-trained on general code. The tier-2 structural ensemble recovers these cases at \textbf{macro F1 0.921} (accuracy 0.864 on 8{,}800 samples), classifying Bernstein--Vazirani at F1 0.82 via CX-gate-pattern features invisible to CodeBERT. Tier-3 rule patterns provide the final fallback for workloads below both ensembles' confidence thresholds, ensuring that every routable input receives at least a deterministic algorithm label.

\subsection{ML Model Component (50\% Weight)}

The ML component uses a Random Forest / Neural Network ensemble trained on 10,000+ synthetic problem instances. The model performs binary classification (Quantum=1, Classical=0) and outputs calibrated probability estimates \cite{simsek2023}.

The synthetic training corpus is generated by AST mutation over twelve base classical-to-quantum algorithm pairs (factorization, search, period-finding, random walk), yielding 14 structural variants per pair via 13 code-level transforms. Feature vectors are z-score standardized; the model is trained with stratified cross-validation and Platt-scaled probabilities, achieving approximately 87\% accuracy on held-out synthetic data (deployed decision-engine latency is reported in Table~\ref{tab:pipeline-timing}). Section~\ref{sec:routing-accuracy} evaluates whether this synthetic-data training transfers to the real-workload benchmark.

\subsection{Rule-Based System (35\% Weight)}

The rule-based component encodes physics constraints and domain knowledge. Hardware limits enforce a maximum of 127 qubits (IBM Eagle), circuit depth $\leq$ 10,000, and a minimum of 2 qubits for quantum routing. Problem-type rules favor quantum execution for factorization (Shor's), search (Grover's), simulation, and optimization (Quantum Approximate Optimization Algorithm (QAOA) \cite{farhi2014}/VQE), while routing sorting, dynamic programming, and matrix operations classically. Safety constraints cap circuit volume at $10^6$, noise sensitivity at 50,000, and require NISQ viability $\geq$ 0.1. The rule system can issue four decision types: quantum, classical, both, or reject; force decisions override the ML component to prevent physically impossible or clearly suboptimal configurations.

\subsection{Cost Analysis Component (15\% Weight)}

The cost analyzer estimates execution time and monetary cost for both quantum and classical execution paths, formalized by \eqref{eq:quantum-cost} and \eqref{eq:classical-cost}.

Gate timing parameters in the quantum execution model are anchored to calibration data retrieved live from IBM Quantum backends via \texttt{QiskitRuntimeService}. Table~\ref{tab:ibm-calibration} summarises the median hardware metrics across the three IBM Quantum backends accessible to this system (retrieved April 2026). The median two-qubit ECR gate time of 68.0\,ns is consistent across all three devices and directly informs the $t_q$ estimation in \eqref{eq:quantum-cost}. The monetary rate $r_{\text{QPU}} = \$1.60/\text{s}$ is taken from IBM's pay-as-you-go schedule \cite{ibm_pricing2024}. As this rate is subject to change, we report the hardware-agnostic QPU-seconds ($t_q$) alongside dollar estimates throughout the paper to allow direct comparison across pricing epochs.

Quantum Cost Model (IBM Quantum Network pay-as-you-go \cite{ibm_pricing2024}):
\begin{equation}
C_q = t_q\, r_{\text{QPU}},
\label{eq:quantum-cost}
\end{equation}
where $C_q$ is the total estimated quantum execution cost, $t_q$ is the estimated QPU reservation time in seconds (the full billable window, including gate execution at the 68.0\,ns median ECR time of Table~\ref{tab:ibm-calibration}, readout, reset, and classical post-processing across all $n_{\text{shots}} = 1024$ measurement shots), and $r_{\text{QPU}} = \$1.60/\text{s}$ is IBM's pay-as-you-go QPU-time rate \cite{ibm_pricing2024}. This is the canonical production pricing model: IBM pay-as-you-go bills purely on QPU-seconds, with no per-shot, per-gate, or job-submission fees. Alternative providers (e.g., AWS Braket) charge per shot and per task rather than per QPU-second, which would add additive terms to \eqref{eq:quantum-cost}; because our benchmark targets the IBM backend family used for real-hardware validation (Table~\ref{tab:hardware-validation}), the IBM pay-as-you-go model is the appropriate reference.

Classical Cost Model (AWS EC2 on-demand pricing \cite{aws_ec2_pricing2024}):
\begin{equation}
C_c = (t_c\, r_{\text{compute}}) + (M_{\text{GB}}\, t_c\, r_{\text{memory}}) + C_{\text{overhead}},
\label{eq:classical-cost}
\end{equation}
where $C_c$ is the total estimated classical execution cost, $t_c$ is the estimated classical execution time in seconds, $r_{\text{compute}} = \$0.0001/\text{s}$ ($\sim$\$0.36/hr, approximating a \texttt{c6i.2xlarge} instance \cite{aws_ec2_pricing2024}) is the compute rate, $M_{\text{GB}}$ is the estimated memory usage in gigabytes, $r_{\text{memory}} = \$0.00001/(\text{GB}\cdot\text{s})$ is the memory rate, and $C_{\text{overhead}} = \$0.001$ is a fixed per-job overhead covering job submission and bookkeeping.

\subsection{Decision Merging}

The three components are merged through a weighted voting algorithm,
\begin{equation}
S_{\text{quantum}} = w_{\text{ML}}\, P_{\text{ML}}^{q} + w_{\text{rules}}\, P_{\text{rules}}^{q} + w_{\text{cost}}\, P_{\text{cost}}^{q},
\label{eq:decision-score}
\end{equation}
where $S_{\text{quantum}}$ is the final quantum suitability score, $P_{\text{ML}}^{q}$ is the ML component's quantum probability, $P_{\text{rules}}^{q}$ is the rule-based component's quantum score, $P_{\text{cost}}^{q}$ is the cost component's quantum preference score, and $w_{\text{ML}} = 0.50$, $w_{\text{rules}} = 0.35$, $w_{\text{cost}} = 0.15$ are the respective component weights.

Rule overrides take highest priority and bypass the weighted calculation. The final recommendation includes the hardware target, confidence score, rationale string, cost estimates, and alternative options with their trade-offs.

\subsection{Feature Importance}

Random Forest Gini-importance analysis shows the ensemble's decisions are driven by physically meaningful features: \texttt{qubits\_required} (0.31) and \texttt{circuit\_volume} (0.24) dominate, reflecting NISQ resource constraints; \texttt{time\_complexity\_encoded} (0.15) and \texttt{entanglement\_factor} (0.11) encode the algorithmic-advantage signal; \texttt{cx\_gate\_ratio} (0.08) implicitly captures two-qubit noise accumulation. Remaining features (\texttt{noise\_sensitivity}, \texttt{gate\_density}, \texttt{nisq\_viability\_score}, \texttt{quantum\_overhead\_ratio}) contribute $\leq$0.04 each. The model has internalized physics-based boundaries rather than surface code syntax.

\subsection{Weight Sensitivity Analysis}

The weights $(w_{\text{ML}}, w_{\text{rules}}, w_{\text{cost}}) = (0.50, 0.35, 0.15)$ are chosen architecturally, not tuned. The ordering $w_{\text{cost}} < w_{\text{rules}} < w_{\text{ML}} = 0.5$ enforces a ``no-dictator'' property: no single component can override a decision alone, and any two of three forms a strict majority (ML+Rules $= 0.85$, ML+Cost $= 0.65$, Rules+Cost $= 0.50$). The cost weight is deliberately small because it is anchored to a specific pricing epoch (Q4~2024 IBM rates); heavier weighting would tie routing decisions to transient prices.

Table~\ref{tab:weight-sensitivity} reports routing counts across six weight configurations on the full 100-workload benchmark. Rule overrides resolve 61 of 100 workloads and bypass the merger entirely, making the weight configuration irrelevant for that majority. Among the 39 workloads that reach the weighted stage, routing is identical to the baseline under the ML-heavy ($70/20/10$), rule-heavy ($30/55/15$), and cost-sensitive ($40/30/30$) configurations, and flips only two under ML-only ($100/0/0$). Only the adversarial equal-weighting ($33/33/34$), which triples the cost weight, produces substantively different routing (26 flips); all 26 flipped workloads have 6--16 qubits---well within the sub-50-qubit regime where classical execution is at least 375$\times$ cheaper (Section~\ref{sec:cost-break-even})---and five of them (QNN variational, VQE caffeine, QAOA supply chain, Shor's $N{=}15$, Shor's $N{=}35$) produced 0.39--5.18\% fidelity on \texttt{ibm\_fez} in Table~\ref{tab:hardware-validation}, empirically confirming the baseline is the correct routing target. We return to the oracle-based accuracy of these weight choices in Section~\ref{sec:routing-accuracy}.

\subsection{Translation Pipeline}

The converter uses a fine-tuned CodeT5-base encoder-decoder model \cite{wang2021} (6+6 layers, $d_{\text{model}}{=}512$, 8 heads, 32,100-token vocabulary) trained on 1,500 curated Python-to-Qiskit pairs covering the six algorithm families used for evaluation (Section~\ref{sec:converter-eval}). The pipeline proceeds in three stages: (i) \emph{logic extraction} parses the input into a Python AST and strips I/O boilerplate, yielding a structural feature vector (loops, branches, iteration depth); (ii) \emph{pattern recognition} matches the extracted logic to known quantum algorithm templates---representative examples include linear search $\rightarrow$ Grover \cite{grover1996}, parity check $\rightarrow$ Deutsch, superposition preparation $\rightarrow$ Hadamard, swap $\rightarrow$ Quantum Swap, hidden bitstring $\rightarrow$ Bernstein--Vazirani, and state transfer $\rightarrow$ Teleportation; (iii) \emph{circuit generation} feeds the matched code as \texttt{"Translate Python to quantum circuit:\textbackslash n\{code\}"}, tokenised at 256 tokens and decoded with beam search ($k{=}3$) up to 300 output tokens, emitting the target Qiskit circuit including gates, measurements, and classical registers.

\subsection{Suitability Scoring}

Each translation candidate receives a three-tier suitability score (HIGH, MEDIUM, LOW) with associated confidence and estimated speedup. Speedup estimates are derived from theoretical complexity comparisons---$O(N)$ classical versus $O(\sqrt{N})$ quantum for Grover \cite{grover1996}, for example, where $N$ is the search-space size---combined with simulator resource metrics from Qiskit Aer execution. Confidence reflects the AST-level overlap between the submitted code and the matched pattern signature, allowing users to judge whether quantum translation offers a meaningful advantage for their specific use case.

\subsection{Evaluation}\label{sec:converter-eval}

The translator was fine-tuned for 7 epochs on the 1{,}500-example Python-to-Qiskit corpus (final training loss 0.5356, validation loss below 0.01 by epoch 2) and evaluated on a 300-sample held-out test set across six algorithm families. Table~\ref{tab:converter-eval} reports the key metrics: the converter produces \textbf{syntactically valid Qiskit circuits on 98.7\%} of inputs and identifies the correct algorithm family on 77.7\%, while the low exact-match rate (EM = 0.197) reflects semantically-equivalent gate orderings rather than incorrect translations.

\section{Hardware Abstraction Layer}\label{sec:hardware-abstraction}

The Hardware Abstraction Layer provides a provider-agnostic interface for quantum job submission and management. Supported providers include IBM, Azure Quantum, and AWS Braket, accessed through a factory-based abstraction that isolates the routing and analysis logic from provider-specific SDKs. The HAL manages the complete job lifecycle:

\begin{enumerate}
    \item \textbf{Device Discovery:} Queries of available devices across all providers, exposing qubit counts, connectivity maps, gate sets, and current status.
    \item \textbf{Job Submission:} Accepts quantum circuits with execution parameters (target device, shot count, transpilation options) and submits through the appropriate provider SDK.
    \item \textbf{Status Tracking:} Monitors job state transitions (queued, running, completed, failed) across providers.
    \item \textbf{Result Retrieval:} Returns measurement distributions, performance metrics, and execution metadata through a unified response format.
\end{enumerate}

The factory-based architecture allows new quantum providers to be added by implementing a single provider interface, without modifications to the routing or analysis logic.

\section{Experimental Evaluation}

\subsection{Benchmark Design}

We evaluated Nexar across 100 synthetic workloads in three categories. The \emph{classical} set (32 workloads) covers deep learning (CNN, transformer, GAN), scientific computing (molecular dynamics, CFD, N-body, PDE), ML (k-means, gradient boosting, reinforcement learning), cryptography (RSA, AES), and combinatorial optimisation (TSP, graph colouring, A*). The \emph{hybrid} set (23 workloads, 0--24 qubits) covers VQE variants (caffeine, job-shop scheduling), QAOA (supply chain, graph partitioning), quantum ML (QSVM, QNN, transfer learning), and error mitigation (ZNE). The \emph{quantum} set (45 workloads, 1--150 qubits) covers molecular simulation (H\textsubscript{2}, LiH, H\textsubscript{2}O, 60-qubit iron-sulfur protein, 120-qubit VQE) \cite{kandala2017}, factoring (Shor's $N{=}15$/$21$/$35$), search (20-qubit Grover, 3-SAT), optimisation (50-qubit QAOA MaxCut, vehicle routing), and foundational algorithms (Deutsch-Jozsa, Bernstein-Vazirani, Simon's, QFT, QPE). All workloads pass through the full Nexar pipeline; analysis, decision, and timing metrics are recorded.

\subsection{Routing Accuracy}\label{sec:routing-accuracy}

\subsubsection{Routing Distribution}
Of the 100 workloads, Nexar routes 79 to classical, 20 to quantum, and rejects 1 as physically infeasible. The rejected workload is a 150-qubit circuit whose state vector cannot be simulated classically ($\sim\!10^{31}$\,MB) and whose qubit count exceeds the 127-qubit IBM~Eagle limit. Quantum-routed workloads span the NISQ-advantage regime and above: 50--120-qubit VQE and QAOA circuits (confidence~1.0), 20--32-qubit optimization and simulation workloads (confidence 0.54--0.80), and a handful of 10--20-qubit hybrids with high superposition or entanglement scores. Every classically-routable workload (qubits\,=\,0) is routed classically with confidence~1.0.

\subsubsection{Dual-Oracle Ground Truth}\label{sec:dual-oracle}
A routing decision can be ``correct'' under two distinct criteria, and Nexar's accuracy depends on which question is asked. We therefore introduce two independently defined oracles that apply literature-cited rules directly to each workload's filename and physical circuit features (qubits, depth, entanglement score), \emph{without referring to Nexar's decision logic}.

\emph{Oracle~A (Algorithmic Advantage)} labels a workload Quantum when a known quantum algorithm provides theoretical speedup in the fault-tolerant regime: classical state-vector simulation infeasible ($\geq\!50$ qubits; \cite{preskill2018}), Shor-class factorization ($\geq\!8$ qubits; \cite{shor1997}), large entangled VQE simulation ($\geq\!20$ qubits, entanglement$\,\geq\!0.3$; \cite{kandala2017}), large QAOA optimization ($\geq\!20$ qubits, entanglement$\,\geq\!0.3$; \cite{farhi2014}), and Grover search ($\geq\!20$ qubits; \cite{grover1996}). It rejects circuits exceeding 127 qubits per \cite{arute2019}.

\emph{Oracle~B (Hardware Viability)} labels a workload Quantum only when current NISQ hardware can produce usable signal. Its thresholds are calibrated to the real-hardware evidence in Table~\ref{tab:hardware-validation}: circuits with post-transpile depth above approximately 200 gates are classified Classical (noise-dominated on \texttt{ibm\_fez}), small-scale circuits below 20 qubits are classified Classical (trivially simulable and free of NISQ overhead), and the sub-50-qubit band with depth $\leq 200$ and entanglement$\,\geq\!0.3$ is the practical NISQ-advantage regime. The memory-wall and rejection thresholds match Oracle~A.

Under Oracle~A, the 100 workloads divide into 85 Classical, 14 Quantum, and 1 Reject; under Oracle~B, 92 Classical, 7 Quantum, and 1 Reject. Every label (100 workloads $\times$ 2 oracles) is reproduced in the supplementary \texttt{oracle\_ground\_truth.csv}.

\subsubsection{Baselines and Ablations}
We compare Nexar against three baselines and three component ablations:
\emph{B1}, a single-line heuristic routing quantum iff qubits $\geq 50$;
\emph{B2}, a circuit-volume heuristic routing quantum iff qubits $\times$ depth $\geq 50{,}000$;
\emph{B3}, a 50/50 random router averaged over 1{,}000 seeds;
\emph{A1}, the ML component in isolation (Random Forest output with no rule or cost influence);
\emph{A2}, the rule-based component in isolation; and
\emph{A3}, the cost analyser in isolation (argmin over classical vs.\ quantum cost estimates).

Table~\ref{tab:routing-accuracy} reports accuracy, precision and recall on the Quantum class, F1$_{(Q)}$, and Cohen's $\kappa$ for every router against both oracles.

\subsubsection{Findings}
Nexar beats random (B3) by more than 30 percentage points on both oracles and achieves Cohen's $\kappa$ of 0.38 (Oracle~A) and 0.48 (Oracle~B), confirming substantive agreement beyond chance. Under Oracle~B, Nexar achieves \emph{perfect recall on the Quantum class} (1.000), meaning every workload that NISQ hardware can usefully execute is routed to quantum; the paper's Table~\ref{tab:hardware-validation} directly corroborates this correspondence.

Three empirical observations emerge. First, Nexar's accuracy is higher under Oracle~B (0.86) than Oracle~A (0.81) because its rule engine applies hardware-viability thresholds (NISQ viability score, noise sensitivity, circuit volume) that mirror Oracle~B more closely than the textbook algorithmic criteria of Oracle~A. This is the correct behaviour for a \emph{present-day} routing platform: cases where Oracle~A and Oracle~B disagree (e.g., Shor's $N=15$, 4-qubit QFT, 20-qubit deep Grover) are workloads where the algorithmic speedup exists in principle but current NISQ hardware cannot deliver it, and Nexar correctly defers these to classical execution. Of the 19 Nexar--Oracle~A disagreements, 18 have direct hardware-evidence counterparts in Table~\ref{tab:hardware-validation}: every sub-20-qubit or deep circuit that Oracle~A labels Quantum produced noise-dominated output (top-outcome fidelity below 10\%) on \texttt{ibm\_fez}, vindicating Nexar's Classical routing.

Second, the \emph{rules-only} ablation (A2) outperforms the full weighted ensemble on both oracles (F1$_{(Q)}$ = 0.636 vs.\ 0.457 on A; 0.667 vs.\ 0.500 on B). This reveals that the ML component, trained on synthetic AST-mutation variants rather than real-hardware execution traces, over-weights algorithmic structure relative to NISQ-viability constraints for circuits in the 10--30 qubit band. The rule engine's deterministic thresholds catch these cases reliably; the ML ensemble adds recall by covering the ALLOW\_BOTH path. A weight-sensitivity sweep across ten configurations in the no-dictator envelope ($w_i \leq 0.5$) produced \emph{identical} accuracy, precision, and F1 metrics: no reweighting of the decision merger moves a single routing decision, because the 61 rule-overridden workloads bypass the merger entirely and the remaining 39 have ML--rule agreement robust to small weight shifts. The practical improvement lever is therefore the rule-engine thresholds, not the weighting scheme, and retraining the ML model on real NISQ execution traces is the primary direction for future work (Section~\ref{sec:future-work}).

Third, the cost-only router (A3) is the weakest component in isolation (F1$_{(Q)}$ below 0.25) because it always prefers the cheaper option, which is classical for every sub-120-qubit workload given current IBM pricing. Cost is useful only as a tie-breaker after ML and rules have converged, which is consistent with the 15\% weight assigned to it in the merger.

\begin{enumerate}

\end{enumerate}

It is important to note that the 50--60 qubit memory wall applies specifically to \emph{full state-vector simulation} of high-entanglement circuits. Approximate classical methods such as Tensor Network simulators, in particular Matrix Product State (MPS) representations, can simulate much larger circuits (100+ qubits) efficiently when the entanglement entropy remains bounded \cite{vinkhuijzen2023}. However, the algorithms central to Nexar's benchmark suite (Shor's factoring, deep VQE ans\"{a}tze with nearest-neighbor CNOT ladders, and QAOA at high circuit depth) generate entanglement that grows with circuit depth, rendering Tensor Network shortcuts inapplicable. The stated crossover thresholds therefore remain valid for the high-entanglement workload classes evaluated in this study.

\subsection{Observations on Current Quantum Limitations}

Benchmark data reflects current NISQ reality: no workload is cheaper on quantum hardware at sub-50-qubit scale (quantum-routed costs \$0.375--\$1.578 vs.\ \$0.001 classical overhead), and the value proposition is forward-looking---quantum advantage materialises only for $\geq$50-qubit exact simulation, fault-tolerant devices, or algorithmic speedups that outgrow the QPU-reservation cost. The routing infrastructure is in place; when hardware crosses the advantage threshold Nexar routes workloads automatically.

\subsection{End-to-End Case Study}

\textbf{120-Qubit VQE $\rightarrow$ Quantum Override.} A 120-qubit VQE ansatz with three variational layers and a nearest-neighbor CNOT ladder is submitted. Analysis extracts \texttt{qubits=120}, \texttt{circuit\_depth=14}, \texttt{cx\_gate\_ratio=0.33}. The physical-necessity rule recognises that classical state-vector simulation would require $2^{120} \times 16$ bytes ($\sim$$10^{31}$\,MB), triggers a \texttt{FORCE\_QUANTUM} override (confidence 1.0) that bypasses ML voting, and the HAL submits the job to a 127-qubit IBM Eagle processor. A contrasting example, a 10-qubit Grover translation of an $O(N)$ linear search over $N{=}1{,}024$ elements with post-transpile depth $\sim$240, is routed to Classical (confidence 0.88): the cost analyser estimates \$0.001 classical versus $\sim$\$0.027 quantum ($240 \cdot 1024 \cdot 68\,\text{ns} \cdot \$1.60/\text{s}$), a $\sim$27$\times$ premium that the $O(\sqrt{N})$ speedup cannot justify at this scale.

\textbf{Hardware-evidence consistency.}
Across all 27 hardware-tested circuits (a subset of the 100-workload benchmark used in Section~\ref{sec:routing-accuracy}), zero routing decisions were contradicted by hardware evidence. Every workload that Nexar routed classically either (a) produced a correct result at low cost confirming that classical execution is sufficient, or (b) produced a noise-dominated result on quantum hardware confirming that it should not have been sent there. No circuit routed classically by Nexar showed a real-hardware result that would have warranted a quantum routing decision.

\textbf{NISQ depth wall.}
Post-transpile circuit depth emerges from the hardware data as the primary predictor of result fidelity, more so than logical qubit count. Circuits with post-transpile depth below approximately 80 gates consistently achieve above 80\% fidelity (DJ constant 99.9\%, BV 96.3\%, QPE 82.3\%). Above depth~$\sim$130, results degrade toward the noise floor. The 4-qubit Grover circuit illustrates this sharply: 4 qubits but depth~241 after \texttt{mcx} decomposition yields only 33.5\% on the target state, while QPE on the same 4 qubits at depth~80 holds 82.3\%. This confirms the physical basis of the NISQ Viability Score used by the rule-based component of the decision engine.

\textbf{Large-circuit verification.}
The 50-qubit QAOA, 60-qubit iron-sulfur VQE, and 120-qubit VQE ansatz were all successfully submitted to and executed on real IBM Quantum hardware, marking the first time these circuits from the benchmark have been executed outside simulation. As expected under single-layer, fixed-parameter ansätze without error mitigation, results are noise-dominated. However, this does not invalidate the quantum routing decision: these circuits are physically impossible to simulate classically (a 120-qubit state vector requires $\sim$$10^{31}$\,MB), so quantum hardware is the only viable execution path regardless of current fidelity. Improving result quality requires error mitigation techniques \cite{kandala2017} or fault-tolerant hardware, both of which are outside the scope of routing.

\textbf{ZNE transpiler insight.}
The ZNE benchmark circuit (24 qubits, logical depth~51) collapsed to post-transpile depth~1 after the Qiskit pass manager recognised that the T/Tdg identity pairs and the forward/reverse GHZ entanglement ladder cancel exactly. The hardware measured $|0\ldots0\rangle$ at 68.5\%, correctly reflecting the near-vacuum state after cancellation. This demonstrates that the Heron~r2 transpiler applies non-trivial circuit identities that can substantially alter the effective depth seen by the hardware, a factor the cost model should account for in future work.

\section{Discussion}

\subsection{Decision Engine Effectiveness}

The hybrid framework proves effective: Nexar beats random by 30 percentage points on both oracles, reaches Cohen's $\kappa\in[0.38,0.48]$, and attains perfect Quantum-class recall under the hardware-viability oracle (Section~\ref{sec:routing-accuracy}). The ablation shows the rules-only component is the single strongest signal (F1$_{(Q)}{=}0.636$ on Oracle~A); the synthetic-data-trained ML ensemble contributes recall but over-weights algorithmic structure in the 10--30 qubit band. Since 61/100 workloads bypass the merger via rule overrides and the remaining 39 are robust to weight perturbation, the main improvement lever is the rule-engine thresholds, with retraining the ML component on real-hardware traces as the longer-term direction (Section~\ref{sec:future-work}). This aligns with the reference architecture of Chinnaraju \cite{chinnaraju2025}; interference-based mixture-of-experts routing \cite{heddad2025} is an alternative as hardware matures.

\subsection{Limitations}

Four limitations bound the present study. First, the ML model is trained on synthetic problem instances rather than real execution traces; retraining on empirical NISQ data should improve routing accuracy as such data becomes available. Second, provider pricing is encoded as static configuration; real-time pricing integration would improve cost accuracy. Third, the current problem-type coverage (optimization, search, simulation, factorization) could be broadened to include quantum chemistry, quantum ML, and cryptographic workloads. Fourth, the system targets current NISQ hardware; fault-tolerant quantum computing will require updated decision logic, cost models, and hardware constraints.

\section{Conclusion and Future Work}\label{sec:future-work}

We presented Nexar, a quantum-classical workload routing platform that integrates automated code analysis, hybrid ML-and-rules-based decision making, and provider-agnostic execution into a single pipeline. Evaluated across 100 workloads against two literature-cited oracles, Nexar achieves 0.81--0.86 accuracy with perfect Quantum-class recall under the hardware-viability oracle, beating random by 30+ percentage points at Cohen's $\kappa = 0.38$--$0.48$. All 27 circuits submitted to \texttt{ibm\_fez} corroborate the routing choices, and the four quantum-routed workloads (20q/50q QAOA, 60q iron-sulfur VQE, 120q VQE) were successfully dispatched to real IBM hardware. Pipeline latency averages 1.5\,s end-to-end with decision inference below 15\,ms, and the cost analysis identifies a clear physical-feasibility crossover at 50--60 qubits beyond which exact classical simulation is no longer possible. Future directions include broader benchmarks on AWS Braket and Azure Quantum, adaptive retraining on empirical execution traces to address the rules-only-beats-ensemble finding, error-mitigation cost modelling, and multi-objective routing over cost, time, accuracy, and carbon footprint.
