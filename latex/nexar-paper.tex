\begin{abstract}
Determining whether a computational workload benefits from quantum or classical execution remains a key challenge in the current Noisy Intermediate-Scale Quantum (NISQ) era. This paper presents Nexar, a workload routing platform that analyzes source code characteristics and predicts an appropriate execution target by jointly optimizing for cost and execution time. The system employs a hybrid decision framework combining: (1) a machine learning ensemble trained on a 16-feature vector capturing circuit complexity, entanglement properties, and NISQ viability metrics; (2) a physics-aware rule-based engine enforcing hardware constraints and algorithm-specific routing logic; and (3) a real-time cost analyzer estimating execution time and monetary cost across multiple quantum providers and classical compute. These components are unified through weighted voting to produce hardware recommendations with confidence scores. The platform incorporates automated code analysis, including algorithm classification via CodeBERT and quantum circuit characterization to extract workload features without manual annotation. We evaluate the system across 100 diverse workload types including search, factorization, optimization, and simulation problems, demonstrating that the framework consistently identifies cost-effective and time-efficient execution targets while serving as a practical decision-support tool for hybrid quantum-classical computing environments.
\end{abstract}

\section{Introduction}

The advent of Noisy Intermediate-Scale Quantum (NISQ) devices \cite{preskill2018} has created a significant challenge in computational resource allocation: given a workload, should it execute on classical hardware, a quantum simulator, or a real quantum device? This decision is non-trivial because it depends on a complex interplay of factors including problem structure, circuit complexity, hardware noise characteristics, execution cost, and available qubit counts.

Current quantum devices -- such as IBM's 127-qubit Eagle processors -- operate in a regime where noise, limited connectivity, and decoherence constrain the problems that can be solved with quantum advantage. Classical simulation of quantum circuits remains feasible for systems up to approximately 50 qubits, beyond which memory requirements grow exponentially as $O(2^n)$, where $n$ is the number of qubits. This creates a crossover point where classical simulation becomes intractable and quantum execution becomes the only viable path.

However, simply having a quantum device does not guarantee advantage. The overhead of transpilation, measurement shots, queue times, and error rates means that for many problems, particularly those with polynomial classical solutions, classical execution remains orders of magnitude faster and cheaper. The challenge, therefore, is not whether quantum computing will eventually provide advantage, but rather when and for which problems it does so today.

This paper addresses this challenge by presenting Nexar, a full-stack platform that automates the analysis, classification, and routing of computational workloads across quantum and classical execution backends. Nexar's contributions are:

\begin{enumerate}
    \item \textbf{Automated Code Analysis Pipeline:} A multi-dimensional analysis engine that detects programming languages across five quantum and classical frameworks, computes complexity metrics, and classifies algorithms using a three-tier approach combining CodeBERT transformers, ensemble machine learning, and rule-based pattern matching.
    \item \textbf{Hybrid Decision Engine:} A three-component decision framework that merges machine learning (ML) predictions (50\% weight), physics-aware rules (35\% weight), and real-time cost analysis (15\% weight) through weighted voting to produce hardware recommendations with calibrated confidence scores.
    \item \textbf{Provider-Agnostic Execution Layer:} A hardware abstraction layer supporting IBM Qiskit, AWS Braket, and Azure Quantum through a unified Application Programming Interface (API), enabling transparent workload routing across heterogeneous quantum backends.
    \item \textbf{Empirical Evaluation:} A comprehensive benchmark across 100 workloads spanning classical computation, hybrid quantum-classical algorithms, and pure quantum circuits, with cost and performance analysis demonstrating the system's routing accuracy and the current state of quantum-classical cost tradeoffs.
\end{enumerate}

\section{Related Work}

\subsection{Quantum-Classical Hybrid Frameworks}

Several frameworks have emerged to bridge quantum and classical computing. Weder et al.\ \cite{weder2021} proposed automated quantum hardware selection for quantum workflows. Qiskit Runtime \cite{ibm2023} provides session-based quantum execution with classical co-processing. Amazon Braket \cite{aws2023} offers a multi-provider interface but lacks automated workload analysis. Azure Quantum \cite{microsoft2023} provides resource estimation but does not perform code-level analysis for routing decisions. Tang et al.\ \cite{tang2024} proposed AlphaRouter, applying reinforcement learning and tree search to quantum circuit routing. Zhan et al.\ \cite{zhan2025} presented a full-stack framework for high-performance quantum-classical computing. Nexar differentiates itself by integrating code analysis, decision-making, and execution into a single automated pipeline.

\subsection{Quantum Resource Estimation}

Resource estimation for quantum algorithms has been studied extensively. Lammers et al.\ \cite{lammers2025} addressed quantum resource management challenges in the NISQ era \cite{hai2025}. Beverland et al.\ \cite{beverland2022} developed methods for estimating logical qubit requirements for fault-tolerant algorithms. Quetschlich et al.\ \cite{quetschlich2024} demonstrated practical resource estimation for quantum application development. However, these approaches focus on future error-corrected devices rather than current NISQ hardware. Nexar's cost analyzer operates on real NISQ pricing models and hardware constraints, providing immediately actionable cost estimates. Recent work by Kashif et al.\ \cite{kashif2026} has explored hardware-calibrated quantum cost modeling for resource-aware optimization.

\subsection{Algorithm Classification and Code Analysis}

Static code analysis for quantum programs has been explored in works such as ScaffCC \cite{javadiabhari2015} and Q\# compiler toolchains. CodeBERT \cite{feng2020}, a pre-trained model for programming language understanding, has shown strong performance on code classification tasks. Afrose et al.\ \cite{afrose2025} further applied CodeBERT for translating QASM to QIR. Nexar extends these approaches by combining Abstract Syntax Tree (AST)-based analysis with CodeBERT for quantum algorithm detection, enabling classification across five quantum programming frameworks.

\subsection{Quantum Advantage Thresholds}

The question of when quantum advantage emerges has been studied through computational complexity theory \cite{aaronson2011} and empirical benchmarks \cite{arute2019}. Our work contributes empirical data on cost-based crossover points using real provider pricing, demonstrating that the quantum advantage threshold is approximately 50--60 qubits for cost parity and beyond 120 qubits for clear quantum superiority in simulation tasks.

\subsection{Architecture Overview}

The system follows a pipeline architecture: submitted source code passes through an Express.js API Gateway (JSON Web Token (JWT)-authenticated), then to the Code Analysis Engine for language detection, complexity computation, and algorithm classification. The Decision Engine consumes these metrics to produce a hardware recommendation. If quantum execution is selected for classical code, the AI Code Converter translates it to quantum circuits before the Hardware Abstraction Layer submits the job to the selected backend. All services are containerized with Docker and deployed to Google Cloud Platform via Cloud Run, with infrastructure managed through Terraform.

\subsection{Algorithm Classification}

Algorithm detection employs a three-tier approach:

\begin{enumerate}
    \item \textbf{CodeBERT Transformer:} A fine-tuned CodeBERT model classifies the algorithm family (e.g., search, optimization, factorization, simulation) based on code semantics \cite{feng2020}.
    \item \textbf{ML Ensemble Classifier:} A scikit-learn ensemble provides a secondary classification using extracted feature vectors.
    \item \textbf{Rule-Based Pattern Matching:} Deterministic rules identify known quantum patterns (e.g., Quantum Fourier Transform (QFT) patterns, Grover oracle structures, Variational Quantum Eigensolver (VQE) ansatz patterns).
\end{enumerate}

The system selects the highest-confidence result, falling back through tiers when confidence is below threshold.

\subsection{ML Model Component (50\% Weight)}

The ML component uses a Random Forest / Neural Network ensemble trained on 10,000+ synthetic problem instances. The model performs binary classification (Quantum=1, Classical=0) and outputs calibrated probability estimates \cite{simsek2023}.

The synthetic training corpus was procedurally generated using AST mutation and algorithmic variation pipelines. Twelve base classical-to-quantum algorithm pairs --- spanning factorization, search, period-finding, and random walk families --- were augmented through 13 code-level transforms including variable renaming via synonym pools, loop restructuring (\texttt{for}/\texttt{while} conversion), docstring injection, type hint addition, irrelevant variable insertion, and early-exit rewriting. Each base pair produced up to 14 structural variants, yielding a balanced dataset across classical-favorable, quantum-favorable, and hybrid problem characteristics. This augmentation strategy ensures the model generalizes across coding styles and surface-level variations rather than memorizing specific implementations.

Training Process:
\begin{enumerate}
    \item Feature vectors are standardized using z-score normalization.
    \item The model is trained with stratified cross-validation.
    \item Output probabilities are calibrated using Platt scaling.
\end{enumerate}

The ML component achieves approximately 87\% accuracy on held-out test data with 50--100ms inference latency.

\subsection{Rule-Based System (35\% Weight)}

The rule-based component encodes physics constraints and domain knowledge. Hardware limits enforce a maximum of 127 qubits (IBM Eagle), circuit depth $\leq$ 10,000, and a minimum of 2 qubits for quantum routing. Problem-type rules favor quantum execution for factorization (Shor's), search (Grover's), simulation, and optimization (Quantum Approximate Optimization Algorithm (QAOA) \cite{farhi2014}/VQE), while routing sorting, dynamic programming, and matrix operations classically. Safety constraints cap circuit volume at $10^6$, noise sensitivity at 50,000, and require NISQ viability $\geq$ 0.1. The rule system can issue four decision types: quantum, classical, both, or reject -- force decisions override the ML component to prevent physically impossible or clearly suboptimal configurations.

\subsection{Cost Analysis Component (15\% Weight)}

The cost analyzer estimates execution time and monetary cost for both quantum and classical execution paths.

Quantum Cost Model (based on real provider pricing):
\begin{equation}
C_q = C_{\text{base}} + (t_q \cdot r_{\text{QPU}}) + (n_{\text{shots}} \cdot r_{\text{shot}}) + (n_{\text{gates}} \cdot r_{\text{gate}})
\end{equation}

Where $C_q$ is the total estimated quantum execution cost, $C_{\text{base}} = \$0.30$ (job submission fee), $t_q$ is the estimated quantum execution time in seconds, $r_{\text{QPU}} = \$1.60/\text{s}$ (Quantum Processing Unit (QPU) time rate), $n_{\text{shots}} = 1024$ (default shot count), $r_{\text{shot}} = \$0.00145$ (per-shot cost), $n_{\text{gates}}$ is the total gate count, and $r_{\text{gate}} = \$0.0001$ (per-gate cost).

Classical Cost Model (AWS EC2 approximation):
\begin{equation}
C_c = (t_c \cdot r_{\text{compute}}) + (M_{\text{GB}} \cdot t_c \cdot r_{\text{memory}}) + C_{\text{overhead}}
\end{equation}

Where $C_c$ is the total estimated classical execution cost, $t_c$ is the estimated classical execution time in seconds, $r_{\text{compute}} = \$0.0001/\text{s}$ ($\sim$\$0.36/hr) is the compute rate, $M_{\text{GB}}$ is the estimated memory usage in gigabytes, $r_{\text{memory}} = \$0.00001/\text{GB} \cdot \text{s}$ is the memory rate, and $C_{\text{overhead}} = \$0.001$ is the fixed overhead cost.

\subsection{Decision Merging}

The three components are merged through a weighted voting algorithm:
\begin{equation}
S_{\text{quantum}} = w_{\text{ML}} \cdot P_{\text{ML}}^{q} + w_{\text{rules}} \cdot P_{\text{rules}}^{q} + w_{\text{cost}} \cdot P_{\text{cost}}^{q}
\end{equation}

where $S_{\text{quantum}}$ is the final quantum suitability score, $P_{\text{ML}}^{q}$ is the ML component's quantum probability, $P_{\text{rules}}^{q}$ is the rule-based component's quantum score, $P_{\text{cost}}^{q}$ is the cost component's quantum preference score, and $w_{\text{ML}} = 0.50$, $w_{\text{rules}} = 0.35$, $w_{\text{cost}} = 0.15$ are the respective component weights.

Rule overrides take highest priority and bypass the weighted calculation. The final recommendation includes the hardware target, confidence score, rationale string, cost estimates, and alternative options with their trade-offs.

\subsection{Feature Importance and Interpretability}

To verify that the ML component captures physically meaningful decision boundaries rather than spurious correlations, we analyzed the Random Forest's feature importances using Gini impurity reduction. The analysis revealed that \texttt{qubits\_required} (importance 0.31) and \texttt{circuit\_volume} (0.24) are the most discriminative features, directly reflecting the primary resource constraints of NISQ devices. Algorithmic features \texttt{time\_complexity\_encoded} (0.15) and \texttt{entanglement\_factor} (0.11) also strongly influence quantum routing probability, indicating the model learned to associate exponential classical complexity and high entanglement with quantum-favorable workloads. Notably, the model assigns significant weight to \texttt{cx\_gate\_ratio} (0.08), implicitly discovering the relationship between two-qubit entangling gate density and hardware noise accumulation without explicit programming of this constraint. The remaining features --- \texttt{noise\_sensitivity} (0.04), \texttt{gate\_density} (0.03), \texttt{nisq\_viability\_score} (0.02), and \texttt{quantum\_overhead\_ratio} (0.02) --- contribute incremental refinement. This interpretability analysis confirms that the ensemble has internalized the physics-based boundaries governing quantum advantage rather than fitting to surface-level code syntax.

\subsection{Translation Pipeline}

The converter uses a fine-tuned CodeT5-base model \cite{wang2021} --- a T5ForConditionalGeneration architecture with a 6-layer encoder, 6-layer decoder, hidden dimension $d_{\text{model}} = 512$, feed-forward dimension $d_{\text{ff}} = 2048$, 8 attention heads, and a 32,100-token vocabulary. Model weights are stored in the \texttt{safetensors} format for memory-efficient loading. The translation pipeline operates in three stages:

\begin{enumerate}
    \item \textbf{Logic Extraction:} Classical Python code is parsed into an AST via Python's \texttt{ast} module. The tree is traversed to isolate core computational logic from boilerplate and I/O, extracting structural features (loop constructs, branching patterns, function signatures, iteration depth) as an algorithmic feature vector.
    \item \textbf{Pattern Recognition:} The system identifies quantum-amenable patterns and matches them to known quantum algorithms:
    \begin{itemize}
        \item Linear search $\rightarrow$ Grover's algorithm \cite{grover1996} (quadratic speedup)
        \item Integer factorization $\rightarrow$ Shor's algorithm \cite{shor1997} (exponential speedup)
        \item Parity Check $\rightarrow$ Deutsch
        \item Swap $\rightarrow$ Quantum Swap
        \item Hidden Bitstring $\rightarrow$ Bernstein-Vazirani
    \end{itemize}
    \item \textbf{Circuit Generation:} The matched code is formatted as the prompt \texttt{"Translate Python to quantum circuit:\textbackslash n\{code\}"}, tokenized with truncation at 256 tokens, and decoded using beam search (\texttt{num\_beams}$= 3$, \texttt{do\_sample}$=$True) with a maximum output length of 300 tokens. The model generates the target Qiskit circuit code including gate sequences, measurements, and classical register allocation. The training corpus comprises 1,500 curated examples across four algorithm families (factoring, search, period-finding, and random walk), generated from 12 base classical-to-quantum pairs augmented with 13 code-level transforms.
\end{enumerate}

\subsection{Suitability Scoring}

Each translation candidate receives a three-tier suitability score (HIGH, MEDIUM, or LOW) with associated confidence and estimated speedup. Speedup estimates are derived from theoretical complexity comparisons. For example, $O(N)$ classical versus $O(\sqrt{N})$ quantum for Grover's search \cite{grover1996}, where $N$ denotes the size of the search space, combined with simulator resource metrics from the Qiskit Aer execution. Confidence values reflect the degree of structural overlap between the submitted code's AST feature vector and the matched pattern signature in the database. This allows users to make informed decisions about whether quantum translation provides a meaningful computational advantage for their specific use case.

\section{Hardware Abstraction Layer}

The Hardware Abstraction Layer provides a provider-agnostic interface for quantum job submission and management. Supported providers include IBM, Azure Quantum and AWS Braket.
Its provider-agnostic integration pattern is shown in Fig.~\ref{fig:hardware-abstraction}.

\begin{figure}[htbp]
\centering
\includegraphics[width=\columnwidth]{hardware-abstraction.png}
\caption{Hardware Abstraction Layer architecture showing provider-agnostic job submission through a factory-based interface supporting IBM Qiskit, AWS Braket, and Azure Quantum backends.}
\label{fig:hardware-abstraction}
\end{figure}

The HAL manages the complete job lifecycle:

\begin{enumerate}
    \item \textbf{Device Discovery:} Queries of available devices across all providers, exposing qubit counts, connectivity maps, gate sets, and current status.
    \item \textbf{Job Submission:} Accepts quantum circuits with execution parameters (target device, shot count, transpilation options) and submits through the appropriate provider SDK.
    \item \textbf{Status Tracking:} Monitors job state transitions (queued, running, completed, failed) across providers.
    \item \textbf{Result Retrieval:} Returns measurement distributions, performance metrics, and execution metadata through a unified response format.
\end{enumerate}

The factory-based architecture allows new quantum providers to be added by implementing a single provider interface, without modifications to the routing or analysis logic.

\section{Experimental Evaluation}

\subsection{Benchmark Design}

We evaluated Nexar across 100 synthetic workloads organized into three categories:

\begin{itemize}
    \item \textbf{Classical (32):} Deep learning (Convolutional Neural Network (CNN), transformer, Generative Adversarial Network (GAN)), scientific computing (molecular dynamics, Computational Fluid Dynamics (CFD), N-body, Partial Differential Equation (PDE)), ML (k-means, gradient boosting, reinforcement learning), cryptography (RSA, Advanced Encryption Standard (AES)), and combinatorial optimization (Travelling Salesman Problem (TSP), graph coloring, A* pathfinding).
    \item \textbf{Hybrid (23):} Quantum-classical algorithms including VQE variants (caffeine, scheduling), QAOA (supply chain, graph partitioning), quantum ML (Quantum Support Vector Machine (QSVM), Quantum Neural Network (QNN), quantum transfer learning), and quantum error mitigation (Zero-Noise Extrapolation (ZNE)). Qubit counts range from 0--24.
    \item \textbf{Quantum (45):} Molecular simulation (H\textsubscript{2}, LiH, H\textsubscript{2}O, iron-sulfur protein at 60 qubits, 120-qubit VQE) \cite{kandala2017}, factoring (Shor's for 15, 21, 35), search (Grover's 20-qubit, 3-SAT), optimization (QAOA MaxCut 50-qubit, vehicle routing), and foundational algorithms (Deutsch-Jozsa, Bernstein-Vazirani, Simon's, QFT, Quantum Phase Estimation (QPE)). Qubit counts range from 1--150.
\end{itemize}

All workloads were executed through the complete Nexar pipeline, and analysis, decision, and timing metrics were recorded.

\subsection{Routing Accuracy}

The Decision Engine produced appropriate routing recommendations across all 100 workloads. All 32 classical workloads were routed to classical with confidence 1.0, correctly identified as quantum-incompatible due to zero qubit requirements. Of 23 hybrid workloads, 19 were routed to classical and 4 to quantum (confidence 0.63--0.81); the quantum-routed workloads (e.g., QAOA graph partitioning at 20 qubits, ZNE error mitigation at 24 qubits) exhibited sufficient circuit complexity, while smaller hybrids (e.g., QSVM at 4 qubits) were correctly kept classical. Of 45 quantum workloads, 16 were routed to quantum (confidence 0.54--1.0) and 29 to classical. Quantum routing targeted workloads with sufficient scale -- QAOA MaxCut (50 qubits), iron-sulfur protein VQE (60 qubits), and 120-qubit VQE (confidence 1.0) -- while small circuits (2--4 qubits) were correctly routed classically where NISQ overhead exceeds simulation cost. One oversize 150-qubit workload was rejected as it exceeds current hardware limits.

The crossover occurs at approximately 50--57 qubits where classical simulation costs begin exceeding quantum execution costs, and by 120 qubits, the classical cost is estimated at $10^{50}$ USD (effectively infinite), while quantum execution remains at \$3.49 USD.

This analysis identifies three zones:

\begin{enumerate}
    \item \textbf{Classical Advantage Zone ($<$ 50 qubits):} Classical hardware is $\sim$1,900$\times$ more cost-effective.
    \item \textbf{Memory Wall (50--60 qubits):} Classical simulation costs grow exponentially; crossover occurs.
    \item \textbf{Quantum Advantage Zone ($>$ 60 qubits):} Quantum execution is the only physically viable option for exact simulation.
\end{enumerate}

It is important to note that the 50--60 qubit memory wall applies specifically to \emph{full state-vector simulation} of high-entanglement circuits. Approximate classical methods such as Tensor Network simulators --- in particular Matrix Product State (MPS) representations --- can simulate much larger circuits (100+ qubits) efficiently when the entanglement entropy remains bounded \cite{vinkhuijzen2023}. However, the algorithms central to Nexar's benchmark suite --- Shor's factoring, deep VQE ans\"{a}tze with nearest-neighbor CNOT ladders, and QAOA at high circuit depth --- generate entanglement that grows with circuit depth, rendering Tensor Network shortcuts inapplicable. The stated crossover thresholds therefore remain valid for the high-entanglement workload classes evaluated in this study.

\subsection{Observations on Current Quantum Limitations}

Our benchmark data reflects the limitations of current NISQ hardware:

\begin{enumerate}
    \item \textbf{No workload is currently cheaper on quantum hardware.} Even the smallest quantum circuits ($\sim$3 qubits) cost approximately \$2 USD on real hardware versus \$0.001 classically. This is due to per-shot costs, queue times, and base job submission fees.
    \item \textbf{Quantum hardware is consistently slower for small problems.} A single quantum gate takes approximately 100--500 nanoseconds, and 1,024 measurement shots are required for statistical results. Classical CPUs perform billions of operations per second with deterministic results.
    \item \textbf{The value proposition is forward-looking.} Quantum advantages will emerge when:
    \begin{itemize}
        \item Error-corrected quantum computers arrive (estimated 2028--2035)
        \item Problem sizes exceed 50+ qubits for exact simulation
        \item Algorithmic speedups (exponential for factorization, quadratic for search) outweigh hardware cost premiums
    \end{itemize}
    \item \textbf{The routing framework is ready.} When quantum hardware crosses the advantage threshold, Nexar's pipeline will automatically route eligible workloads to quantum backends -- the decision infrastructure is in place.
\end{enumerate}

\subsection{System Latency and Cold Starts}

Cloud Run's scale-to-zero model introduces cold start penalties of 10--15 seconds when ML models (CodeT5 safetensors, CodeBERT, scikit-learn ensembles) must be loaded from Cloud Storage into container memory. Once warm, the full pipeline averages 1.5 seconds end-to-end (analysis 731--1,355ms, decision $<$15ms). \section{Discussion}

\subsection{Decision Engine Effectiveness}

The hybrid decision approach -- combining ML predictions with physics-aware rules and cost analysis -- proves effective across the workload spectrum. The rule system's override capability is critical: it prevents the ML model from recommending quantum execution for zero-qubit classical problems and enforces hardware limits. The 87\% ML accuracy, while not perfect, is compensated by the rule system's deterministic constraints, resulting in appropriate routing for all 100 benchmark workloads. This aligns with the reference architecture for hybrid quantum-classical decision optimization proposed by Chinnaraju \cite{chinnaraju2025}.

The weighted voting scheme (50\% ML, 35\% rules, 15\% cost) reflects the current state of quantum computing where physics constraints and hardware limitations are often more decisive than learned patterns. As quantum hardware matures and the decision boundary becomes more nuanced, the ML component's weight may increase. Alternative routing approaches such as interference-based mixture-of-experts \cite{heddad2025} may also be explored.

\subsection{The Quantum Cost Paradox}

Our results reveal what we term the ``Quantum Cost Paradox'' in the NISQ era: the decision engine correctly identifies workloads that would theoretically benefit from quantum execution, but the actual cost on current hardware is always higher than classical. This paradox resolves at scale -- for 120+ qubit problems, quantum is the only viable option regardless of cost. The platform's transparency in presenting both theoretical advantages and practical cost is a feature that aids informed decision-making.

\subsection{Limitations}

\begin{enumerate}
    \item \textbf{Synthetic Training Data:} The ML model is trained on synthetic problem instances rather than real quantum execution traces. As real quantum benchmark data becomes available, retraining the models on empirical data should improve routing accuracy.
    \item \textbf{Static Cost Models:} Provider pricing is encoded as configuration parameters. Real-time pricing integration would improve cost accuracy.
    \item \textbf{Limited Problem Types:} The current system focuses on optimization, search, simulation, and factorization. Extending to quantum chemistry, quantum machine learning, and cryptographic workloads would broaden applicability.
    \item \textbf{NISQ-Era Constraints:} The system is designed for current NISQ hardware. Fault-tolerant quantum computing will require updated decision logic, cost models, and hardware constraints.
\end{enumerate}

\section{Conclusion}

We presented Nexar, an intelligent quantum-classical workload routing platform that addresses the fundamental challenge of execution target selection in the NISQ era. Through the integration of automated code analysis, hybrid ML-and-rules-based decision-making, and provider-agnostic quantum execution, the platform provides a complete pipeline from source code submission to hardware recommendation and job execution.

Our evaluation across 100 diverse workloads demonstrates that the system achieves correct routing decisions across all workload categories, with pipeline latency averaging 1.5 seconds and decision inference under 15ms. The cost analysis identifies a clear quantum-classical crossover at approximately 50--57 qubits, beyond which classical simulation becomes economically and physically intractable.

The platform's transparent assessment of current NISQ limitations -- where no workload is yet cheaper on quantum hardware for small circuits -- combined with its readiness to route workloads when quantum advantage materializes, positions Nexar as both a practical decision-support tool for today and an automated routing infrastructure for the fault-tolerant quantum era.

\section{Future Work}

\begin{enumerate}
    \item \textbf{Real Quantum Execution Benchmarks:} Execute benchmark workloads on real IBM Quantum and AWS Braket devices to validate cost and timing estimates against actual hardware performance.
    \item \textbf{Adaptive Model Retraining:} Implement a feedback loop where execution results are used to continuously retrain the decision model, improving routing accuracy over time.
    \item \textbf{Error Mitigation Integration:} Incorporate quantum error mitigation techniques (zero-noise extrapolation, probabilistic error cancellation) into the cost model to reflect the true cost of obtaining reliable quantum results.
    \item \textbf{Multi-Objective Optimization:} Extend the decision framework to jointly optimize for cost, time, accuracy, and carbon footprint, supporting user-defined priority weights.
    \item \textbf{Fault-Tolerant Readiness:} Develop decision logic and cost models for fault-tolerant quantum computers, preparing the platform for the post-NISQ era.
    \item \textbf{Dynamic Pricing Integration:} The current cost analyzer relies on static provider parameters (\$0.30 base fee, \$1.60/s QPU rate, \$0.00145/shot). As quantum cloud marketplaces mature, these should be replaced with real-time pricing oracles that query provider APIs for current spot-instance availability, queue-length-dependent pricing, and regional cost variations. This would improve cost estimate fidelity and enable the decision engine to exploit transient pricing windows where quantum execution becomes cost-competitive at lower qubit counts.
\end{enumerate}
