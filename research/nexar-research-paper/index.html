    NEXAR: Intelligent Quantum-Classical Workload Routing Platform¶

    Prasad H G A T$^1$, Jayasinghe Y L$^1$, Hettiarachchi S R$^1$, Siriwardhana A H L T S$^1$

    $^1$ Sri Lanka Institute of Information Technology (SLIIT)

    Abstract¶

    Determining whether a computational workload benefits from quantum or classical execution remains a key challenge in the current Noisy Intermediate-Scale Quantum (NISQ) era. This paper presents Nexar, an intelligent workload routing platform that analyzes source code characteristics and predicts the optimal execution target by jointly optimizing for cost and execution time. The system employs a hybrid decision framework combining: (1) a machine learning ensemble trained on a 16-feature vector capturing circuit complexity, entanglement properties, and NISQ viability metrics; (2) a physics-aware rule-based engine enforcing hardware constraints and algorithm-specific routing logic; and (3) a real-time cost analyzer estimating execution time and monetary cost across multiple quantum providers and classical compute. These components are unified through weighted voting to produce hardware recommendations with confidence scores. The platform incorporates automated code analysis --- including algorithm classification via CodeBERT and quantum circuit characterization --- to extract workload features without manual annotation. We evaluate the system across 30 diverse workload types including search, factorization, optimization, and simulation problems, demonstrating that the framework consistently identifies cost-effective and time-efficient execution targets while serving as a practical decision-support tool for hybrid quantum-classical computing environments.

    Keywords: quantum computing, workload routing, NISQ, decision engine, hybrid computing, cost analysis, CodeBERT, quantum advantage

    1. Introduction¶

    The advent of Noisy Intermediate-Scale Quantum (NISQ) devices [1] has created an unprecedented challenge in computational resource allocation: given a workload, should it execute on classical hardware, a quantum simulator, or a real quantum device? This decision is non-trivial because it depends on a complex interplay of factors including problem structure, circuit complexity, hardware noise characteristics, execution cost, and available qubit counts.

    Current quantum devices --- such as IBM's 127-qubit Eagle processors --- operate in a regime where noise, limited connectivity, and decoherence constrain the problems that can be solved with quantum advantage. Classical simulation of quantum circuits remains feasible for systems up to approximately 50 qubits, beyond which memory requirements grow exponentially as $O(2^n)$. This creates a crossover point where classical simulation becomes intractable and quantum execution becomes the only viable path.

    However, simply having a quantum device does not guarantee advantage. The overhead of transpilation, measurement shots, queue times, and error rates means that for many problems --- particularly those with polynomial classical solutions --- classical execution remains orders of magnitude faster and cheaper. The challenge, therefore, is not whether quantum computing will eventually provide advantage, but rather when and for which problems it does so today.

    This paper addresses this challenge by presenting Nexar, a full-stack platform that automates the analysis, classification, and routing of computational workloads across quantum and classical execution backends. Nexar's contributions are:

    1. Automated Code Analysis Pipeline: A multi-dimensional analysis engine that detects programming languages across five quantum and classical frameworks, computes complexity metrics, and classifies algorithms using a three-tier approach combining CodeBERT transformers, ensemble machine learning, and rule-based pattern matching.

    2. Hybrid Decision Engine: A three-component decision framework that merges ML predictions (50% weight), physics-aware rules (35% weight), and real-time cost analysis (15% weight) through weighted voting to produce hardware recommendations with calibrated confidence scores.

    3. Provider-Agnostic Execution Layer: A hardware abstraction layer supporting IBM Qiskit, AWS Braket, and Azure Quantum through a unified API, enabling transparent workload routing across heterogeneous quantum backends.

    4. Empirical Evaluation: A comprehensive benchmark across 30 workloads spanning classical computation, hybrid quantum-classical algorithms, and pure quantum circuits, with cost and performance analysis demonstrating the system's routing accuracy and the current state of quantum-classical cost tradeoffs.

    2. Related Work¶

    2.1 Quantum-Classical Hybrid Frameworks¶

    Several frameworks have emerged to bridge quantum and classical computing. Qiskit Runtime [2] provides session-based quantum execution with classical co-processing. Amazon Braket [3] offers a multi-provider interface but lacks automated workload analysis. Azure Quantum [4] provides resource estimation but does not perform code-level analysis for routing decisions. Nexar differentiates itself by integrating code analysis, decision-making, and execution into a single automated pipeline.

    2.2 Quantum Resource Estimation¶

    Resource estimation for quantum algorithms has been studied extensively. Beverland et al. [5] developed methods for estimating logical qubit requirements for fault-tolerant algorithms. However, these approaches focus on future error-corrected devices rather than current NISQ hardware. Nexar's cost analyzer operates on real NISQ pricing models and hardware constraints, providing immediately actionable cost estimates.

    2.3 Algorithm Classification and Code Analysis¶

    Static code analysis for quantum programs has been explored in works such as ScaffCC [6] and Q# compiler toolchains. CodeBERT [7], a pre-trained model for programming language understanding, has shown strong performance on code classification tasks. Nexar extends these approaches by combining AST-based analysis with CodeBERT for quantum algorithm detection, enabling classification across five quantum programming frameworks.

    2.4 Quantum Advantage Thresholds¶

    The question of when quantum advantage emerges has been studied through computational complexity theory [8] and empirical benchmarks [9]. Our work contributes empirical data on cost-based crossover points using real provider pricing, demonstrating that the quantum advantage threshold is approximately 50--60 qubits for cost parity and beyond 120 qubits for clear quantum superiority in simulation tasks.

    3. System Architecture¶

    Nexar is implemented as a microservices monorepo comprising six independently deployable services communicating through a central API Gateway. The architecture is designed for modularity, allowing each component to scale independently and be updated without affecting the overall system.

    3.1 Architecture Overview¶

    The system follows a pipeline architecture with the following request flow:

    1. Code Submission: Users submit source code through a React-based frontend.
    2. Authentication and Routing: An Express.js API Gateway authenticates requests via JWT and routes them to backend services.
    3. Code Analysis: The Code Analysis Engine performs multi-dimensional analysis including language detection, complexity computation, and algorithm classification.
    4. Routing Decision: The Decision Engine consumes analysis metrics and produces a hardware recommendation.
    5. Code Conversion (optional): If quantum execution is selected and the code is classical, the AI Code Converter translates it to quantum circuits.
    6. Execution: The Hardware Abstraction Layer submits jobs to the selected quantum or classical backend.

    3.2 Service Specifications¶

    Service Stack Port Function
    Frontend React 18, TypeScript, Vite, Tailwind CSS 5173 User interface, visualization
    API Gateway Express 5, TypeScript 3000 Authentication, proxying, logging
    Code Analysis Engine FastAPI, Python, CodeBERT, tree-sitter 8002 Language detection, metrics, classification
    Decision Engine FastAPI, Python, scikit-learn, XGBoost 8003 ML-based routing prediction
    AI Code Converter FastAPI, Python, Transformers 8001 Classical-to-quantum translation
    Hardware Abstraction Layer FastAPI, Python, Qiskit, Braket SDK 8004 Multi-provider job execution

    All services are containerized with Docker and deployed to Google Cloud Platform via Cloud Run, with infrastructure managed through Terraform.

    4. Code Analysis Engine¶

    The Code Analysis Engine serves as the analytical foundation of the pipeline, transforming raw source code into a structured feature set that downstream components consume.

    4.1 Language Detection¶

    The engine supports five programming languages and frameworks: Python, Qiskit, Cirq, Q#, and OpenQASM. Detection uses an ML-based classifier achieving confidence scores above 0.85 across all evaluated workloads. In our benchmark, average language detection confidence was 0.955 (sd = 0.028) across 30 workloads.

    4.2 Classical Metrics Extraction¶

    For all detected code, the engine computes classical software metrics using tree-sitter-based AST parsing:

    • Cyclomatic Complexity: Measures the number of independent execution paths. Values in our benchmark ranged from 2 to 31.
    • Cognitive Complexity: Assesses human comprehension difficulty, ranging from 0 to 39 across workloads.
    • Time Complexity Classification: Inferred from loop structure and recursion patterns --- classes include $O(1)$, $O(n)$, $O(n \log n)$, $O(n^2)$, $O(n^3)$, $O(n^k)$, and $O(2^n)$.
    • Space Complexity: Estimated from data structure allocation patterns.
    • Structural Metrics: Loop count, conditional count, function count, and nesting depth.

    4.3 Quantum Metrics Extraction¶

    For quantum code, additional metrics are extracted:

    • Qubit Requirements: The number of qubits needed for circuit execution.
    • Circuit Depth: The number of sequential gate layers, directly impacting decoherence.
    • Gate Count and CX Gate Ratio: Total gates and the proportion of two-qubit entangling gates (CNOT/CX), which are the primary source of noise on NISQ devices.
    • Superposition Score (0--1): Quantifies the degree of superposition in the circuit.
    • Entanglement Score (0--1): Measures entanglement density, a key indicator of quantum advantage potential.

    4.4 Algorithm Classification¶

    Algorithm detection employs a three-tier approach:

    1. CodeBERT Transformer: A fine-tuned CodeBERT model classifies the algorithm family (e.g., search, optimization, factorization, simulation) based on code semantics.
    2. ML Ensemble Classifier: A scikit-learn ensemble provides a secondary classification using extracted feature vectors.
    3. Rule-Based Pattern Matching: Deterministic rules identify known quantum patterns (e.g., QFT patterns, Grover oracle structures, VQE ansatz patterns).

    The system selects the highest-confidence result, falling back through tiers when confidence is below threshold.

    5. Decision Engine¶

    The Decision Engine is the central intelligence of Nexar. It implements a hybrid prediction system combining three components through weighted voting.

    5.1 Feature Vector Design¶

    The engine operates on a 16-feature input vector composed of 10 raw features and 6 derived physics features:

    Raw Features (10):

    # Feature Description
    1 problem_size Number of elements or variables
    2 qubits_required Quantum bits needed
    3 circuit_depth Sequential quantum gate layers
    4 gate_count Total quantum gates
    5 cx_gate_ratio Proportion of entangling gates (0--1)
    6 superposition_score Superposition potential (0--1)
    7 entanglement_score Entanglement density (0--1)
    8 memory_requirement_mb Classical RAM required
    9 problem_type_encoded Categorical: factorization, search, optimization, etc.
    10 time_complexity_encoded Complexity class: exponential, polynomial, etc.

    Derived Physics Features (6):

    # Feature Formula Purpose
    11 Circuit Volume $qubits \times depth$ Quantum resource intensity
    12 Noise Sensitivity $qubits \times depth \times cx_ratio$ Error accumulation prediction
    13 Quantum Overhead Ratio $problem_size / qubits$ Encoding efficiency
    14 NISQ Viability Score Composite penalty function Feasibility on current hardware
    15 Gate Density $gate_count / qubits$ Operational complexity per qubit
    16 Entanglement Factor $cx_ratio \times entanglement$ Combined entanglement potential

    The NISQ Viability Score applies penalties when qubits exceed 50, circuit depth exceeds 1,000, or circuit volume exceeds 50,000, reflecting the practical limits of current hardware.

    5.2 ML Model Component (50% Weight)¶

    The ML component uses a Random Forest / Neural Network ensemble trained on 10,000+ synthetic problem instances. The model performs binary classification (Quantum=1, Classical=0) and outputs calibrated probability estimates.

    Training Process: 1. Feature vectors are standardized using z-score normalization. 2. The model is trained with stratified cross-validation. 3. Output probabilities are calibrated using Platt scaling.

    The ML component achieves approximately 87% accuracy on held-out test data with 50--100ms inference latency.

    5.3 Rule-Based System (35% Weight)¶

    The rule-based component encodes physics constraints and domain knowledge:

    Hardware Capability Constraints: - Maximum qubits: 127 (IBM Eagle processor limit) - Maximum circuit depth: 10,000 - Minimum qubits for quantum routing: 2

    Problem-Type Routing Rules: - Quantum-favorable: Factorization (Shor's, exponential speedup), Search (Grover's, quadratic speedup), Simulation (natural quantum simulation), Optimization (QAOA/VQE) - Classical-favorable: Sorting ($O(n \log n)$ classical optimality), Dynamic Programming (sequential dependencies), Matrix Operations (mature classical libraries)

    Safety Constraints: - Maximum circuit volume: 1,000,000 - Maximum noise sensitivity: 50,000 - Minimum NISQ viability: 0.1

    The rule system can issue four decision types: FORCE_QUANTUM, FORCE_CLASSICAL, ALLOW_BOTH, or REJECT. Force decisions override the ML component, ensuring that physically impossible or clearly suboptimal configurations are not recommended.

    5.4 Cost Analysis Component (15% Weight)¶

    The cost analyzer estimates execution time and monetary cost for both quantum and classical execution paths.

    Quantum Cost Model (based on real provider pricing):

    $$C_q = C_{base} + (t_q \cdot r_{QPU}) + (n_{shots} \cdot r_{shot}) + (n_{gates} \cdot r_{gate})$$

    Where $C_{base} = \$0.30$ (job submission), $r_{QPU} = \$1.60/s$, $r_{shot} = \$0.00145$, $r_{gate} = \$0.0001$, and $n_{shots} = 1024$ (default).

    Classical Cost Model (AWS EC2 approximation):

    $$C_c = (t_c \cdot r_{compute}) + (M_{GB} \cdot t_c \cdot r_{memory}) + C_{overhead}$$

    Where $r_{compute} = \$0.0001/s$ (~$0.36/hr), $r_{memory} = \$0.00001/GB \cdot s$, and $C_{overhead} = \$0.001$.

    5.5 Decision Merging¶

    The three components are merged through a weighted voting algorithm:

    $$S_{quantum} = w_{ML} \cdot P_{ML}^{q} + w_{rules} \cdot P_{rules}^{q} + w_{cost} \cdot P_{cost}^{q}$$

    where $w_{ML} = 0.50$, $w_{rules} = 0.35$, $w_{cost} = 0.15$.

    Rule overrides (FORCE_QUANTUM, FORCE_CLASSICAL) take highest priority and bypass the weighted calculation. The final recommendation includes the hardware target, confidence score, rationale string, cost estimates, and alternative options with their trade-offs.

    6. AI Code Converter¶

    The AI Code Converter enables the platform to accept classical Python code and automatically translate it into quantum circuits when quantum execution is recommended.

    6.1 Translation Pipeline¶

    The converter uses a transformer-based code generation model to produce quantum circuit code targeting Qiskit or Cirq frameworks. The translation pipeline operates in three stages:

    1. Logic Extraction: Core computational logic is isolated from boilerplate, I/O, and classical scaffolding code.
    2. Pattern Recognition: The system identifies quantum-amenable patterns and matches them to known quantum algorithms:
    3. Linear search $\rightarrow$ Grover's algorithm (quadratic speedup)
    4. Combinatorial optimization $\rightarrow$ QAOA (potential exponential speedup)
    5. Integer factorization $\rightarrow$ Shor's algorithm (exponential speedup)
    6. Circuit Generation: The transformer model generates the target quantum circuit with appropriate gate sequences, measurements, and classical post-processing.

    6.2 Suitability Scoring¶

    Each translation candidate receives a suitability score (HIGH, MEDIUM, or LOW) with associated confidence and estimated speedup. This allows users to make informed decisions about whether the quantum translation provides meaningful advantage.

    7. Hardware Abstraction Layer¶

    The Hardware Abstraction Layer (HAL) provides a provider-agnostic interface for quantum job submission and management.

    7.1 Supported Providers¶

    Provider SDK Capabilities
    IBM Quantum Qiskit 127-qubit Eagle processors, simulators
    AWS Braket Amazon Braket SDK IonQ, Rigetti, OQC devices
    Azure Quantum Azure Quantum SDK IonQ, Quantinuum devices

    7.2 Job Lifecycle Management¶

    The HAL manages the complete job lifecycle:

    1. Device Discovery: Queries available devices across all providers, exposing qubit counts, connectivity maps, gate sets, and current status.
    2. Job Submission: Accepts quantum circuits with execution parameters (target device, shot count, transpilation options) and submits through the appropriate provider SDK.
    3. Status Tracking: Monitors job state transitions (queued, running, completed, failed) across providers.
    4. Result Retrieval: Returns measurement distributions, performance metrics, and execution metadata through a unified response format.

    The factory-based architecture allows new quantum providers to be added by implementing a single provider interface, without modifications to the routing or analysis logic.

    8. Experimental Evaluation¶

    8.1 Benchmark Design¶

    We evaluated Nexar across 30 workloads organized into three categories:

    • Classical workloads (10): Neural network backpropagation, protein folding energy computation, molecular dynamics, genome sequence alignment, PCA dimensionality reduction, epidemic SIR modeling, FFT signal processing, k-means clustering, heat equation PDE solving, and decision tree classification.
    • Hybrid workloads (5): Quantum SVM kernel estimation, quantum Boltzmann sampling, variational quantum eigensolver (excited states), quantum GAN generator, and drug molecule screening.
    • Quantum workloads (15): $H_2$ molecule VQE, LiH molecule UCCSD, Ising model simulation, Heisenberg spin chain, Shor's factoring, QAOA portfolio optimization, variational quantum classifier, quantum walk on graphs, VQE water molecule, quantum error correction, GHZ state teleportation, Deutsch-Jozsa (8-qubit), quantum phase estimation, HHL algorithm, and 120-qubit VQE molecular simulation.

    All workloads were executed through the complete Nexar pipeline, and analysis, decision, and timing metrics were recorded.

    8.2 Routing Accuracy¶

    The Decision Engine produced correct routing recommendations across all workload categories:

    Classical Workloads (10/10 correct): All 10 classical workloads were routed to classical hardware with confidence 1.0. The rule system correctly identified these as quantum-incompatible due to zero qubit requirements (rationale: "Too few qubits (0 < 2)"), demonstrating that the safety constraints prevent unnecessary quantum execution.

    Hybrid Workloads (5/5 appropriate): The hybrid workloads exhibited split routing. Three workloads with sufficient quantum characteristics (Boltzmann sampling, VQE excited states, quantum GAN) were routed to quantum hardware with confidence 0.85. The QSVM kernel estimation (4 qubits) was routed to classical despite quantum characteristics, reflecting the cost-aware decision that small circuits run more efficiently classically. Drug molecule screening was correctly identified as classical-dominant.

    Quantum Workloads (15/15 appropriate): Of 15 quantum workloads, 10 were routed to quantum hardware (confidence 0.85--1.0) and 5 were routed to classical. The classical routing for small circuits (3--4 qubits: GHZ teleportation, HHL, $H_2$ VQE, quantum walk) reflects the correct assessment that NISQ overhead makes small quantum circuits more expensive than classical simulation. The 120-qubit VQE was routed to quantum with confidence 1.0 --- the only workload where classical execution is physically intractable.

This reflects the current state of NISQ economics where per-shot costs, QPU time charges, and base job fees dominate.

    8.4 Cost Break-Even Analysis¶

    Despite the current quantum cost premium, analysis of the cost scaling behavior reveals a fundamental crossover point driven by the exponential memory requirements of classical quantum simulation.

    For an $n$-qubit quantum state, classical simulation requires $2^n$ complex amplitudes, consuming approximately $2^n \times 16$ bytes of memory. This creates the following scaling:

    Qubits Classical Memory Classical Feasibility
    20 16 MB Trivial
    30 16 GB Desktop
    40 16 TB Cluster
    50 16 PB Supercomputer limit
    60 16 EB Physically impossible
    120 $\sim 2 \times 10^{31}$ MB ---

    The crossover occurs at approximately 50--57 qubits where classical simulation costs begin exceeding quantum execution costs, and by 120 qubits, the classical cost is estimated at $10^{50}$ USD (effectively infinite), while quantum execution remains at $3.49 USD.

    This analysis identifies three zones: 1. Classical Advantage Zone (< 50 qubits): Classical hardware is ~1,900x more cost-effective. 2. Memory Wall (50--60 qubits): Classical simulation costs grow exponentially; crossover occurs. 3. Quantum Advantage Zone (> 60 qubits): Quantum execution is the only physically viable option for exact simulation.

    8.6 Observations on Current Quantum Limitations¶

    Our benchmark data honestly reflects the limitations of current NISQ hardware:

    1. No workload is currently cheaper on quantum hardware. Even the smallest quantum circuits ($\sim$3 qubits) cost approximately $2 USD on real hardware versus $0.001 classically. This is due to per-shot costs, queue times, and base job submission fees.

    2. Quantum hardware is consistently slower for small problems. A single quantum gate takes approximately 100--500 nanoseconds, and 1,024 measurement shots are required for statistical results. Classical CPUs perform billions of operations per nanosecond with deterministic results.

    3. The value proposition is forward-looking. Quantum advantage will emerge when:

    4. Error-corrected quantum computers arrive (estimated 2028--2035)
    5. Problem sizes exceed 50+ qubits for exact simulation

    6. Algorithmic speedups (exponential for factorization, quadratic for search) outweigh hardware cost premiums

    7. The routing framework is ready. When quantum hardware crosses the advantage threshold, Nexar's pipeline will automatically route eligible workloads to quantum backends --- the decision infrastructure is in place.

    9. Deployment and Infrastructure¶

    Nexar is deployed on Google Cloud Platform using a fully automated infrastructure-as-code approach. All six microservices run as independently scalable containers on Cloud Run, with Docker images stored in Google Artifact Registry. Infrastructure is defined declaratively in Terraform, organized across configuration files for Cloud Run services (cloudrun.tf), IAM policies (iam.tf), container registries (artifact_registry.tf), and parameterized variables (variables.tf).

    CI/CD is handled through GitHub Actions workflows that lint, test, build, and deploy each service independently. Pull requests trigger preview deployments, while merges to the main branch trigger production deployments. ML models and analysis artifacts are stored in Cloud Storage and loaded at service startup.

    10. Discussion¶

    10.1 Decision Engine Effectiveness¶

    The hybrid decision approach --- combining ML predictions with physics-aware rules and cost analysis --- proves effective across the workload spectrum. The rule system's override capability is critical: it prevents the ML model from recommending quantum execution for zero-qubit classical problems and enforces hardware limits. The 87% ML accuracy, while not perfect, is compensated by the rule system's deterministic constraints, resulting in appropriate routing for all 30 benchmark workloads.

    The weighted voting scheme (50% ML, 35% rules, 15% cost) reflects the current state of quantum computing where physics constraints and hardware limitations are often more decisive than learned patterns. As quantum hardware matures and the decision boundary becomes more nuanced, the ML component's weight may increase.

    10.2 The Quantum Cost Paradox¶

    Our results reveal what we term the "Quantum Cost Paradox" in the NISQ era: the decision engine correctly identifies workloads that would theoretically benefit from quantum execution, but the actual cost on current hardware is always higher than classical. This paradox resolves at scale --- for 120+ qubit problems, quantum is the only viable option regardless of cost. The platform's transparency in presenting both theoretical advantage and practical cost is a feature that aids informed decision-making.

    10.3 Limitations¶

    1. Synthetic Training Data: The ML model is trained on synthetic problem instances rather than real quantum execution traces. As real quantum benchmark data becomes available, retraining on empirical data should improve routing accuracy.
    2. Static Cost Models: Provider pricing is encoded as configuration parameters. Real-time pricing integration would improve cost accuracy.
    3. Limited Problem Types: The current system focuses on optimization, search, simulation, and factorization. Extending to quantum chemistry, quantum machine learning, and cryptographic workloads would broaden applicability.
    4. NISQ-Era Constraints: The system is designed for current NISQ hardware. Fault-tolerant quantum computing will require updated decision logic, cost models, and hardware constraints.

    11. Conclusion¶

    We presented Nexar, an intelligent quantum-classical workload routing platform that addresses the fundamental challenge of execution target selection in the NISQ era. Through the integration of automated code analysis, hybrid ML-and-rules-based decision-making, and provider-agnostic quantum execution, the platform provides a complete pipeline from source code submission to hardware recommendation and job execution.

    Our evaluation across 30 diverse workloads demonstrates that the system achieves correct routing decisions across all workload categories, with pipeline latency averaging 1.5 seconds and decision inference under 15ms. The cost analysis identifies a clear quantum-classical crossover at approximately 50--57 qubits, beyond which classical simulation becomes economically and physically intractable.

    The platform's honest assessment of current NISQ limitations --- where no workload is yet cheaper on quantum hardware for small circuits --- combined with its readiness to route workloads when quantum advantage materializes, positions Nexar as both a practical decision-support tool for today and an automated routing infrastructure for the fault-tolerant quantum era.

    12. Future Work¶

    1. Real Quantum Execution Benchmarks: Execute benchmark workloads on real IBM Quantum and AWS Braket devices to validate cost and timing estimates against actual hardware performance.
    2. Adaptive Model Retraining: Implement a feedback loop where execution results are used to continuously retrain the decision model, improving routing accuracy over time.
    3. Error Mitigation Integration: Incorporate quantum error mitigation techniques (zero-noise extrapolation, probabilistic error cancellation) into the cost model to reflect the true cost of obtaining reliable quantum results.
    4. Multi-Objective Optimization: Extend the decision framework to jointly optimize for cost, time, accuracy, and carbon footprint, supporting user-defined priority weights.
    5. Fault-Tolerant Readiness: Develop decision logic and cost models for fault-tolerant quantum computers, preparing the platform for the post-NISQ era.
